## 1. 系统架构总览

```
┌─────────────────────────────────────────────────────────┐
│               Web 前端 (Vue 3 + Element Plus)            │
│  首页看板 │ 用例生成 │ 审核标注 │ 知识库(卡片式)          │

┌────────────────────────┴────────────────────────────────┐
│               后端服务 (Python/FastAPI)                   │
│  ┌───────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ │
│  │ 知识库CRUD│ │ 检索(混合)│ │ LLM生成  │ │ 审核统计  │ │
│  └───────────┘ └──────────┘ └──────────┘ └──────────┘ │
│  ┌──────────────────────────────────────────────────┐  │
│  │  六大测试技术Prompt (等价类/边界值/决策表/       │  │
│  │  状态迁移/错误推测/组合测试) + 覆盖率总结        │  │
│  └──────────────────────────────────────────────────┘  │
│  ┌──────────────────────────────────────────────────┐  │
│  │  知识检索(关键词+ChromaDB向量) + 自我修正        │  │
│  └──────────────────────────────────────────────────┘  │
└────────────────────────┬────────────────────────────────┘
                         │
┌────────────────────────┴────────────────────────────────┐
│                     数据层                                │
│  ┌──────────┐ ┌──────────┐ ┌────────────────────────┐  │
│  │  SQLite  │ │ ChromaDB │ │    LLM API (OpenAI兼容)       │  │
│  │(知识库)  │ │ (向量库)  │ │                         │  │
│  └──────────┘ └──────────┘ └────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
```

### 核心设计原则

**知识先行，Prompt后行** — 任何用例生成请求，必须先经过知识检索，将检索结果注入Prompt。不允许直接"裸调"LLM。

**可追溯** — 每条生成的用例都要记录它引用了哪些知识条目。用户审核时能回溯"这个预期结果是从哪条业务规则推导出来的"。

**瘦启动** — 先用SQLite、单进程跑通闭环，后期再根据需要切PostgreSQL、加Redis缓存、做服务拆分。

## 2. 核心模块设计

### 2.1 知识库管理模块（系统基石）

知识库是整个平台的护城河。设计中定义了 **7种知识类型**（4种结构化 + 3种向量化）：

#### 类型1：字段字典
```
字段名称 | 页面展示名 | 数据来源 | 类型 | 合法取值/枚举 | 业务含义 | 所属模块
channel_count | 设备数    | camera表  | int  | ≥0           | 一个摄像机可含多通道，页面写"设备数"实际是"通道数" | 实时视频
```
**用途**：根治"字段幻觉"——生成用例前，对照字段字典校验每个引用的字段是否存在、类型是否正确。

#### 类型2：业务规则
```
规则名称 | 规则表达式                | 规则类型 | 所属模块 | 来源
VIP免运费 | user_level='vip' → freight=0 | 硬规则   | 订单模块 | PRD v3.2 + 代码逻辑
```
规则类型分：
- **硬规则**：代码强制约束（如状态机、字段取值）
- **软规则**：业务惯例（如"大促期间优惠券不可叠加"）

**用途**：根治"规则幻觉"——约束LLM只能在规则允许的空间内生成用例。

#### 类型3：状态机
```
实体   | 状态流转                     | 所属模块
订单   | 待支付→已支付→已发货→已完成  | 订单模块
订单   | 待支付→已取消                | 订单模块
订单   | 已支付→已退款→已关闭         | 订单模块
```
**用途**：根治"上下文缺失"——生成用例时注入状态机，让LLM知道前置状态依赖。

#### 类型5：字段-术语映射
```
页面术语 | 技术字段      | 映射关系     | 所属模块
设备数   | channel_count | 1摄像头=N通道 | 实时视频
```
**用途**：解决"设备数 vs 通道数"这类命名歧义问题。

#### 类型6：PRD文档库（文件上传 + 向量化）

支持上传 PDF/Word/Markdown/TXT 格式的 PRD 文档，自动文本提取后分块向量化存入 ChromaDB。生成用例时，从 PRD 库中检索最相关的文档片段注入 Prompt，让 LLM 具备完整业务上下文。

```
文件名称 | 所属模块 | 文件格式 | 提取文本 | 向量化分块 | 上传时间
```

**用途**：根治"上下文缺失"——PRD 里有详细业务描述和隐含规则，直接检索注入比人工提炼更完整。

#### 类型7：缺陷记录库（历史Bug）

```
缺陷ID | 标题 | 严重级别 | 根因分类 | 描述 | 关联模块 | 关联用例 | 发生时间
```

每条缺陷的描述文本向量化存入 ChromaDB。生成用例时，检索历史上最相关的缺陷记录，注入 Prompt 让 LLM 知道"**这里曾经出过什么问题**"，针对性地生成回归测试用例。

**用途**：根治"历史遗漏"——老测试知道哪些地方容易出 Bug，现在 AI 通过缺陷记录也能知道。

### 2.2 用例生成模块（核心引擎）

生成流程：

```
用户输入需求
  ├─ 粘贴文本 ──→ 直接进入流程
  └─ 上传PRD文件 ──→ 格式检测+文本提取 → 可选LLM关键信息识别 → 进入流程
       │
       ▼
┌──────────────────┐
│ 需求解析           │ ← 提取关键实体（模块名、字段名、操作类型）
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ 知识检索           │ ← 混合检索：关键词匹配（字段字典/业务规则/状态机）
│                    │            + 语义检索（历史用例向量相似度 Top-K）
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ 约束验证           │ ← 自动校验：引用字段是否在字典中存在？
│                    │   预期结果是否与业务规则冲突？
│                    │   前置条件是否满足状态机流转？
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ Prompt 构造        │ ← 组装系统提示（角色 + 约束规则 + 格式要求）
│                    │   + 检索到的知识上下文
│                    │   + 用户原始需求
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ LLM 生成           │ ← 调用 LLM API，流式返回
└──────┬───────────┘
       │
       ▼
┌──────────────────┐
│ 后处理 + 追溯注入  │ ← 为每条用例打上知识引用标签
│                    │   去掉冗余/重复项
└──────┬───────────┘
       │
       ▼
   用例结果列表
```

#### Prompt 构造策略（三层结构）

```
[System Layer]
你是一个专业的功能测试用例设计专家。你必须严格基于以下知识库信息生成用例，不得编造任何字段、规则或状态：

## 可用字段字典
{table: field_dictionary}

## 强制业务规则（必须遵守）
{table: business_rules}

## 状态机流转（不可违反）
{table: state_machines}

## 术语映射（注意不要被页面命名误导）
{table: term_mapping}

[Context Layer]
## 相似历史用例（供参考格式和粒度）
{retrieved_historical_cases}

[User Request Layer]
{用户输入的需求描述}

输出要求：每条用例包含 模块、场景、用例标题、前置条件、测试步骤（序号+操作+数据）、预期结果。以JSON数组格式输出。
```

### 2.3 审核与反馈闭环模块

```
生成用例列表
     │
     ▼
逐条人工审核 ──→ 标注 [可用] ──→ 进入可用用例池
     │
     ▼
  标注 [不可用] ──→ 选归因类型：
                    │ 字段幻觉 → 提示补充字段字典
                    │ 规则幻觉 → 提示补充业务规则
                    │ 上下文缺失 → 提示补充状态机
                    │ 风格不一致 → 标记为反例
                    │
                    ▼
              系统记录归因 → 引导用户补充对应知识条目
                              │
                              ▼
                    知识库增量更新 → 下次生成质量提升
```

**关键指标自动计算**：
- 可用率 = 标注[可用]数 / 总用例数
- 幻觉率分布 = 各类归因占比
- 知识库覆盖度 = 被引用的知识条目 / 总知识条目

### 2.4 PRD 文件解析模块

用户可以直接上传PRD文档，系统自动提取文本内容作为生成输入。支持格式：

| 格式 | 解析方式 | 说明 |
|---|---|---|
| PDF | PyPDF2 / pdfplumber | 提取文本内容，保留段落结构 |
| Word (.docx) | python-docx | 提取段落、表格 |
| Markdown (.md) | 直接读取 | 保留标题层级，用于后续按章节生成 |
| TXT | 直接读取 | 纯文本输入 |

```
用户上传 PRD 文件 (PDF/Word/MD/TXT)
         │
         ▼
┌────────────────────┐
│ 格式检测 + 文本提取  │
└────────┬───────────┘
         │
         ▼
┌────────────────────┐
│ 关键信息自动提取     │ ← 从PRD中识别模块名、字段名、规则描述、状态流转描述
│ (可选：LLM辅助解析)  │   可作为知识库候选条目提示用户录入
└────────┬───────────┘
         │
         ▼
  提取后的文本 + 用户确认 → 进入用例生成流程（关联知识库检索）
```

**与粘贴文本的区别**：上传PRD文件后，系统可以额外做「PRD关键信息提取」——用LLM从PRD中自动识别出可能涉及的字段、规则线索，与已有知识库做对比，如果有知识库未覆盖的，给出补充建议。这个环节不阻塞流程，是可选的增强步骤。

### 2.5 Excel 导入/导出模块

**导入**：支持三种Sheet模板
| Sheet名 | 内容 | 说明 |
|---|---|---|
| 字段字典 | 字段名/页面名/类型/枚举/含义 | 必选 |
| 业务规则 | 规则名/表达式/类型/来源 | 必选 |
| 历史用例 | 标题/前置/步骤/预期/模块 | 可选但强烈推荐 |

**导出**：生成用例导出为Excel，兼容团队现有模板格式（需用户提供样例模板）。

## 3. 技术选型建议

| 层 | 选型 | 理由 |
|---|---|---|
| 前端 | Vue 3 + Element Plus | 生态成熟、表格/表单组件丰富、适合B端、用户指定 |
| 后端 | Python + FastAPI | 与AI/LLM生态无缝对接、开发效率高 |
| 关系数据库 | SQLite（开发）→ PostgreSQL（生产） | SQLite零配置起步；PostgreSQL支持pgvector扩展 |
| 向量数据库 | ChromaDB | 轻量、Python原生、与LangChain集成好、适合嵌入式场景 |
| LLM SDK | 自封装调用（OpenAI 兼容格式） | 与Qwen API格式一致，方便切换 |
| Excel处理 | openpyxl | Python Excel处理的事实标准 |
| PRD解析 | pdfplumber + python-docx | PDF文本提取 + Word文档解析 |
| 部署 | Docker + docker-compose | 一键启动前后端+数据库 |

**为什么不选LangChain/LlamaIndex**：MVP阶段逻辑清晰，自建检索+Prompt构造足够，引入框架反而增加调试成本和复杂度。后续如果需要复杂RAG策略（Rerank、多路召回融合），再考虑引入。

## 4. 数据模型概要

```sql
-- 当前核心表

knowledge_bases -- 知识库 (id, name, description, created_at, updated_at)
field_dicts     -- 字段字典 (id, kb_id, field_name, display_name, data_type, enum_values, description)
business_rules  -- 业务规则 (id, kb_id, rule_name, rule_type[hard|soft], expression, description, source)
state_machines  -- 状态机 (id, kb_id, entity, from_state, to_state, condition)
term_mappings   -- 术语映射 (id, kb_id, ui_term, tech_field, mapping_desc)
prd_documents   -- PRD文档 (id, kb_id, filename, file_format, raw_text, chunk_count, created_at)
defect_records  -- 缺陷记录 (id, kb_id, title, severity, root_cause, description, related_case, occurred_at, created_at)
test_cases      -- 用例 (id, title, precondition, steps, expected_result, source[manual|ai], quality_score, knowledge_refs[json], batch_id, req_text, created_at)
review_records  -- 审核记录 (id, case_id, status[approved|rejected], reject_reason, reviewer_comment, reviewed_at)

-- ChromaDB collections:
--   prd_docs         -- PRD文档分块向量化
--   historical_cases -- 历史用例向量化
--   defect_records   -- 缺陷描述向量化
```

## 6. 关键设计决策

### 6.1 向量检索 + 关键词检索的混合策略
- **关键词匹配**：用于字段字典、业务规则、状态机——这些是精确匹配场景，"订单状态"不能模糊成"订单情况"
- **语义检索**：用于历史用例——"下单失败"应能匹配到"支付异常"的用例，用向量相似度
- **混合权重**：关键词结果优先展示，语义结果作为few-shot补充

### 6.2 知识条目引用追溯
每条AI生成的用例在Prompt阶段要求LLM输出 `knowledge_refs` 字段：
```json
{
  "title": "VIP用户下单免运费",
  "knowledge_refs": [
    {"type": "business_rule", "id": "br_001", "name": "VIP免运费"},
    {"type": "field_dict", "id": "fd_012", "name": "freight_amount"}
  ]
}
```
这样审核时用户能快速定位"这条用例的依据是对是错"。

### 6.3 约束检查层（生成前自动校验）
在构造Prompt之前，系统可以自动做两层检查：
1. **字段存在性检查**：需求中提到的字段是否在字段字典中？
2. **状态可达性检查**：如果需求涉及状态变更，是否在状态机中存在合法路径？

如果检查不通过，先生成"知识缺失预警"，提示用户先补充知识再用例生成。
